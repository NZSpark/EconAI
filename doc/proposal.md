# EconAI 需求文档

## 1. 项目概述

### 1.1 项目定位

EconAI 是面向经济政策研究机构的人工智能工具包，通过结合大语言模型能力、可信证据库、项目上下文和透明推理链，加速高质量经济政策分析和研究成果的交付。

### 1.2 核心差异化

- **信息质量控制**：基于精心筛选的数据源（内部报告、项目文件夹、结构化知识库）构建，而非依赖不受控的网络内容
- **可审计性**：所有输出附带逐句级别的来源溯源，在政策环境中可追溯、可审核
- **安全合规**：混合 LLM 部署架构（Claude API 云端 + 本地模型私有化），满足等保二级和 GDPR 合规要求
- **机构级协作**：支持角色权限 + 项目组隔离的组织架构

### 1.3 关键指标

| 指标 | 描述 |
|------|------|
| 可靠性 | 输出断言可追溯到具体源文档的段落/页码 |
| 安全性 | 敏感数据不出机构内网，通过本地 LLM 处理；满足等保二级 + GDPR |
| 易用性 | Web 界面操作，无需编程背景即可完成完整分析流程 |
| 规模 | 支撑 5 万+ 文档索引检索，< 20 并发分析师 |
| 交付 | 1-2 月交付 MVP 原型，覆盖全部四个核心分析场景 |

## 2. 目标用户

### 2.1 用户画像

EconAI 面向**政策研究机构的整体部署**，主要用户角色：

| 角色 | 描述 | 典型场景 |
|------|------|----------|
| **政策分析师** | 日常使用的主力用户，负责撰写政策草案、综述、对比分析 | 上传文献、配置知识库、生成分析报告 |
| **高级研究员** | 审核分析结果，把控学术质量 | 审查来源标注、校验关键论断、批准报告发布 |
| **项目管理员** | 管理项目组、成员权限、知识库资源 | 创建项目组、配置知识库源、分配用户角色 |
| **系统管理员** | 运维部署、安全管理 | 配置 LLM 后端、监控系统运行、审计日志 |

### 2.2 使用方式

- 通过 **Web 浏览器** 访问，无需安装客户端
- 分析师的核心工作流：创建项目 → 配置知识源 → 提交分析任务 → 审核输出 → 导出成果
- 非技术用户可完成完整流程，高级用户可通过提示词定制分析深度和风格

## 3. 核心功能场景

### 3.1 文献综述

**输入**：多篇 PDF/Word 格式的政策文献、研究报告、学术论文
**过程**：
1. 自动解析文档结构（章节、摘要、结论、参考文献）
2. 提取核心论点、方法论、数据来源、政策建议
3. 识别不同文献之间的观点异同、证据冲突
4. 按主题聚类，标注每篇文献的方法论质量和证据强度
**输出**：
- 结构化文献综述报告（.docx / Markdown）
- 文献对比矩阵表（.xlsx）
- 逐句来源标注

### 3.2 政策草案撰写

**输入**：机构内部报告、历史政策文件、相关法规文件、当前议题说明
**过程**：
1. 从知识库中检索与当前议题相关的政策背景、历史沿革、已有方案
2. 按照政策文件标准结构（背景、问题分析、方案建议、实施路径、预算评估）生成初稿
3. 标注每个关键段落的数据来源和参考依据
**输出**：
- 政策文件初稿（.docx）
- 引用清单
- 简报 PPT（可选）

### 3.3 政策选项对比

**输入**：需要对比的多个政策选项（可以是文档，也可以是结构化描述）
**过程**：
1. 对每个选项从多个维度拆解：（经济影响、社会影响、实施难度、时间周期、财政成本、风险、证据支持度）
2. 检索知识库中与各选项相关的证据和支持材料
3. 生成多维度对比分析，标注证据强度（强/中/弱/无证据）
**输出**：
- 政策选项对比分析报告（.docx）
- 多维度对比矩阵表（.xlsx）
- 关键差异摘要 PPT

### 3.4 技术资料解读

**输入**：复杂技术报告（如环境影响评估、宏观计量模型结果、行业技术白皮书）
**过程**：
1. 解析技术文档中的专业术语、方法论、数据模型
2. 提取与当前政策议题相关的关键发现
3. 将技术语言翻译为政策语言，标注简化过程中不精确的地方
**输出**：
- 结构化解读报告（.docx / Markdown）
- 术语对照表
- 关键发现摘要 PPT

## 4. 知识库设计

### 4.1 双轨知识库架构

| 类型 | 说明 | 生命周期 |
|------|------|----------|
| **项目知识库** | 分析师为特定任务临时导入的文件集合（PDF、Word、Excel 等） | 随项目创建，项目结束后归档 |
| **机构知识库** | 机构长期维护的结构化文档库，按主题和类型索引 | 持续更新，跨项目复用 |

### 4.2 支持的文档格式

| 类别 | 格式 | 处理方式 |
|------|------|----------|
| 文档 | PDF、Word (.docx)、纯文本 (.txt)、Markdown (.md) | 全文解析+结构化提取 |
| 数据 | Excel (.xlsx)、CSV | 表格解析+语义理解 |
| 简报 | PowerPoint (.pptx) | 逐页解析 |
| 邮件 | .eml | 元数据提取+正文解析 |
| 网页 | Web 存档 (.mhtml/.html) | 正文提取+原始链接保留 |
| 扫描件 | 图片 PDF/图片文件 | OCR 预处理后进入解析流程 |

### 4.3 知识库操作

- **导入**：拖拽上传、文件夹批量导入、API 接入
- **索引**：自动分块、嵌入向量生成、元数据提取
- **检索**：语义搜索 + 关键词搜索混合检索，支持按项目/文档类型/时间范围过滤
- **管理**：文档版本管理、过期标记、权限控制

## 5. 输出与引用体系

### 5.1 输出格式

- **Markdown**：结构化文本，方便二次编辑和版本管理
- **Word (.docx)**：符合 GB/T 9704 公文国标格式的正式政策文档，含封面、版头、主体、版记等标准元素
- **Excel (.xlsx)**：数据表格、对比矩阵
- **PowerPoint (.pptx)**：关键发现简报

### 5.2 逐句溯源系统

每个 LLM 生成的断言如果基于知识库中的某篇文献，需标注：
- 源文档名称
- 页码范围/段落编号
- 相关原文摘录（可选展示）
- 置信度标记（直接引用 / 推理总结 / 跨文献综合）

**用户交互**：
- 在 Web 界面中点击任意输出句子，展开查看来源详情
- 导出 .docx 时以脚注或尾注形式保留引用
- 引用可开关显示（阅读模式 / 审核模式）

## 6. 权限与安全

### 6.1 权限模型

```
组织
├── 项目组 A (知识库、分析结果隔离)
│   ├── 分析师 A1 (读写)
│   ├── 分析师 A2 (读写)
│   └── 高级研究员 A3 (读写+审核)
├── 项目组 B
│   └── ...
└── 系统管理员 (全局管理)
```

### 6.2 隔离规则

- 项目组间知识库和分析结果互不可见
- 高级研究员可查看本组所有成员的工作成果
- 跨组共享需项目管理员显式授权
- 机构知识库可按组配置访问权限

### 6.3 合规要求

| 标准 | 定级 | 关键要求 | 实现方式 |
|------|------|----------|----------|
| 等保 | 二级 | 网络安全、数据安全、访问控制、审计日志 | 私有化部署、传输加密、RBAC、操作审计、定期自查 |
| GDPR | 正式适用 | 合法性基础、数据最小化、目的限制、删除权、数据可携带、DPIA | 用户数据管理功能、数据导出/删除 API、数据处理协议、隐私影响评估、数据保护官（DPO）对接 |

#### 等保二级要点

- 物理安全：机构机房已有的物理防护
- 网络安全：内网部署、防火墙、TLS 传输加密
- 主机安全：操作系统加固、访问控制
- 应用安全：身份认证、RBAC、审计日志、输入校验
- 数据安全：存储加密、备份恢复、数据完整性校验

#### GDPR 合规要点

- 数据处理合法性基础：用户知情同意 + 合同履行
- 数据主体权利：提供数据访问、更正、删除、导出功能
- 数据最小化：仅采集和分析必要数据
- 跨境传输：如使用云端 API（Claude），确认数据传输合规机制（SCC 或等效保障）
- DPIA（数据保护影响评估）：系统上线前完成
- 数据处理记录：维护完整的处理活动记录（ROPA）

## 7. 技术架构

### 7.1 整体架构

```
┌─────────────────────────────────────────────┐
│               Web 前端 (React)               │
├─────────────────────────────────────────────┤
│            API 网关 (FastAPI)                 │
├──────┬──────┬──────┬──────┬────────────────┤
│ 文档  │ 知识库│ 任务  │ 输出  │  用户/权限      │
│ 解析  │ 管理  │ 编排  │ 生成  │  管理          │
├──────┴──────┴──────┴──────┴────────────────┤
│              LLM 路由层                      │
├──────────────────┬─────────────────────────┤
│  本地 LLM (敏感)  │  云端 API (非敏感)        │
├──────────────────┴─────────────────────────┤
│        向量数据库 + 文档存储                  │
└─────────────────────────────────────────────┘
```

### 7.2 关键模块

| 模块 | 功能 |
|------|------|
| 文档解析引擎 | 多格式解析、OCR、结构化提取、分块 |
| 知识库管理 | 向量索引、元数据管理、检索排序 |
| LLM 路由 | 按任务敏感度分流到本地或云端模型 |
| 任务编排 | 多步骤分析工作流（检索→生成→引用校验→格式化） |
| 输出生成 | 逐句溯源标注、多格式导出 |
| 权限管理 | RBAC、项目组隔离、审计日志 |

### 7.3 LLM 混合部署

> **重要说明**：Anthropic Claude 是目前确认选用的云端 LLM。Claude 不支持本地私有化部署，因此敏感数据的本地推理需搭配其他可私有化部署的模型。以下为混合部署方案：

| 数据类型 | 部署位置 | 模型 | 说明 |
|----------|----------|------|------|
| 内部报告、政策文件、敏感数据 | 本地 GPU 集群 | Qwen / DeepSeek / Llama（待评测选定） | 基于机构已有 GPU 集群部署，数据不出内网 |
| 公开文献、非敏感任务 | 云端 API | **Claude API**（已确认） | 利用 Claude 的分析推理能力处理非敏感任务 |
| 最终审核、敏感上下文 | 本地 GPU 集群 | 同本地模型 | 审核环节严格本地执行 |

**本地模型选型建议**（需在实际 GPU 硬件上评测后确定）：
1. **Qwen3**：国产模型，中文政策文本理解能力强，合规性好
2. **DeepSeek-V3**：综合能力强，推理成本低，中文表现优异
3. **Llama 4 + 中文微调**：开源生态成熟，部署工具链完善

路由规则：
- 新建分析任务时系统自动判断：仅依赖公开知识源 → Claude API；涉及内部文档 → 本地模型
- 用户可手动指定单次任务使用的模型
- 管理员可配置默认路由策略和敏感度阈值

### 7.4 技术选型

| 层次 | 技术 | 说明 |
|------|------|------|
| 前端 | **React + TypeScript**（已确认） | Web 界面，响应式设计，企业级组件库 |
| 后端 | Python + FastAPI | 高性能异步 API |
| 数据库 | PostgreSQL | 业务数据存储 |
| 向量数据库 | Milvus / Qdrant | 知识库向量检索，支撑 5 万+ 文档规模 |
| 文档存储 | MinIO / 本地文件系统 | 原始文档存储 |
| 文档解析 | Unstructured + 自研适配 | 多格式解析 |
| OCR | **Tesseract + 中文语言包**（已确认） | 开源 OCR，成本低且支持私有化 |
| 本地 LLM | vLLM / Ollama | 本地模型推理服务，利用已有 GPU 集群 |
| 云端 LLM | **Anthropic Claude API**（已确认） | 非敏感任务的主力分析引擎 |
| 任务队列 | Celery + Redis | 异步长任务处理 |
| 监控 | Prometheus + Grafana | 系统运行监控 |

### 7.5 基础设施

- **GPU 集群**：机构已有，可直接用于本地 LLM 推理部署
- **存储**：5 万+ 文档的原始存储 + 向量索引，预估总存储约 5-10 TB
- **网络**：内网部署，通过反向代理对外提供 Web 访问

## 9. 非功能需求

### 9.1 性能

- 单篇文档解析时间 < 30 秒（50 页 PDF 参考）
- 分析任务响应时间 < 3 分钟（典型文献综述，10 篇文献）
- Web 界面首屏加载 < 3 秒
- 知识库支持 5 万+ 文档的索引和检索，检索延迟 < 2 秒
- 并发用户数：< 20 分析师同时在线，峰值任务队列深度 50

### 9.2 可用性

- 系统可用性 > 99%（私有化部署由机构运维保障）
- 长任务异步执行，支持进度展示和失败重试
- 操作日志完整可审计

### 9.3 安全性（等保二级 + GDPR）

- 数据传输全链路 TLS 1.2+ 加密
- 静态数据 AES-256 加密存储
- 完整的身份认证（支持 LDAP/SSO 对接机构账号体系）
- RBAC 细粒度访问控制
- API 访问频率限制
- 所有操作审计日志保留不少于 6 个月
- 敏感数据标记与自动路由：标注为内部/敏感的知识源自动路由到本地 LLM
- GDPR 数据主体权利：提供数据访问、更正、删除、导出功能

| F | 首批知识库文档清单 | 确认 MVP 阶段需导入的机构知识库文档范围和访问权限 | 业务负责人 |

## 附录 A：术语表

| 术语 | 定义 |
|------|------|
| 知识库 | 用于支撑 LLM 检索增强生成（RAG）的结构化文档集合 |
